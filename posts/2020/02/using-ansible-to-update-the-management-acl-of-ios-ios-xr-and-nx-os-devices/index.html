Using Ansible to update the management ACL (access-list) on Cisco IOS, IOS-XR and NX-OS devices

Scenario: You have a large brownfield environment that is a mixture of Cisco IOS, IOS-XR and NX-OS devices. The configuration of these devices has been previously managed manually and as such, there is no standard configuration. Each of the operating systems has a slightly different syntax for making changes and each device’s access-list for management access could be any name or number and be either a standard or extended access-list.

You have been tasked with updating this management access-list to include a new Host or Subnet. To manually do this it would require days, if not weeks, of your time and would be a mundane process that would likely lead to errors or devices missed. Instead you decide to develop an Ansible role that automates this configuration for you.

The Ansible inventory

Our environment follows one of Ansible’s best practice directory layouts. The top level of the directory would contain a separate production and development inventory directory, with each having its own group_vars/host_vars.

inventories/
   production/
      hosts              # inventory file for production network
      group_vars/        # here we assign variables to particular groups
         all.yml
         ios.yml
         iosxr.yml
         nxos.yml
      host_vars/         # here we assign variables to particular systems
         ioshost1.yml
         iosxrhost1.yml
         nxoshost1.yml

   development/
      hosts               # inventory file for development network
      group_vars/
         group1.yml       # here we assign variables to particular groups
         group2.yml
      host_vars/
         devhost1.yml     # here we assign variables to particular systems
         devhost2.yml

playbook_1.yml
playbook_2.yml
playbook_3.yml

roles/
    role1/
        tasks/
        handlers/
        files/
        templates/

The production inventory file categorises each device by operating system and location.

1. Initializing the new role

From the root of your Ansible directory, change to the roles directory and create the new role:

cd roles/
ansible-galaxy init management_acl

The role is initialized using the ansible-galaxy init command. This builds an empty role with the below base structure:

roles/
    management_acl/
        tasks/
        handlers/
        files/
        templates/
        vars/
        defaults/
        meta/

2. Creating the configuration tasks

Now that we have our base role created in ansible/roles/ we need to write the configuration tasks within the role for each Cisco operating system (IOS, IOS-XR and NXOS). This is done within the tasks directory of the role.

cd roles/management_acl/tasks/
touch ios_config.yml ios-xr_config.yml nxos_config.yml

The code for this role and the above configuration tasks can be found here. But to summarise, a combination of show commands and regex will be used in each OS specific configuration file to perform the following tasks:

  1. Use the Ansible modules ios_command, iosxr_command or nxos_command to issue the relevant show command required to discover the management access-list name. All output from the show command will be registered as the variable “name".
  2. Use Regex to parse the variable name.stdout and register the management access-list name/number as the variable “aclNAME".
  3. Fail for any device with no management access-list configured, do not execute any further tasks and print a debug message to the terminal.
  4. Use the same Ansible modules from Step 1 and the “aclNAME” variable from Step 2 to view the management access-list configuration. This also shows the access-list type and all output from the show command will be registered as the variable “type".
  5. Use Regex to parse the variable type.stdout and register the management access-list type (standard | extended) as the variable “aclTYPE".

For every device we now have the management access-list name (aclNAME) and type (aclTYPE) registered as variables that can be used to apply the new configuration.”

  1. Use the Ansible modules ios_config, iosxr_config or nxos_config to: 6.1 Remove the explicit deny statement (if present). 6.2 Add the new Host or Subnet entry. 6.3 Add the explicit deny statement with logging.

  2. Use the same Ansible modules from Step 6 to resequence the management access-list (start at 10, increment by 10) and save the configuration.

3. Creating the playbook

The playbook is used to call the role we have developed and specifies some playbook level parameters such as the hosts to target and the connection type to use.

---

- name: PLAY - Management ACL Playbook
  hosts: ios,iosxr,nxos                     # Uses a device (or group) in our inventory file
  gather_facts: no                          # Disables fact gathering for the above hosts
  connection: network_cli                   # Used for network devices with a limited execution environment
  
  vars:
    cfg_acl:
          - no deny any
          - permit {{ acl }}
          - deny any log

  roles:
    - management_acl

Executing the Playbook

The playbook can then be invoked from your terminal by using the ansible-playbook command and passing the playbook file:

cph@cph:~$ ansible-playbook management_acl -i inventories/production/
